# 第 2 章 —— 闭环中的 AI

"Auto 是 Rust 的脚本层"这个故事围绕 AI 来写代码展开。本章讲*为什么脚本模式是它的正确基底*——这不是营销，是延迟。

## AI 实际跑的循环

AI 写代码时，它不是写一次就停。它写、运行、读输出、发现遗漏的边界情况、重写、再运行。那个循环就是生产力的单位，其成本是每次迭代的时间。

- **直接写 Rust**：每一轮都付编译税。对一个不简单的 crate，那是几十秒到几分钟，*每个 AI 轮次*。AI 花在等 `cargo build` 上的时间比思考还多。
- **写 Auto**：AutoVM 即时解释——没有编译步骤。AI 调 `/api/run`，毫秒级拿到 stdout，立即修正。

编辑上面的代码块：改 `sample` 字符串，或加一个新的棘手输入，然后点 **Run in VM**。那个亚秒级的往返就是全部要点。AI 可以快速地错很多次，因为犯错很便宜。

## 为什么"脚本"不等于"Python"

Python 迭代也快。那为什么不直接让 AI 写 Python、之后再重写成 Rust？因为那个重写才是昂贵的部分——一个完整的第二工程，AI 得把动态、无类型、GC 的语义翻译成静态、所有权跟踪、零成本抽象的 Rust。两种语言在"程序是什么意思"上不一致，所以翻译是设计问题，不是机械问题。

Auto 与 Rust 在语义上一致。上面的 `count_words` 函数，其类型、控制流、整数语义与 a2r 将产出的 Rust 完全相同。点 **Transpile to Rust** 看看：没有留下任何重写要做。迭代基底与发布基底是同一门语言。

## 诚实的一面

脚本模式容忍 AI 的不完美，恰恰*因为*代码在迭代中是可丢弃的：一次错误的尝试只付一次亚秒级重跑，什么都没提交。存活下来的是最终的 Auto 源码，而这份源码正是 a2r 转成可审计 Rust 的东西。AI 不被信任负责发布——转译器和 Rust 编译器才负责。AI 被信任负责原型，在原型阶段快速失败是特性。
