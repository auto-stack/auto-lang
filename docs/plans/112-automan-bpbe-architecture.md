# Plan 112: AutoMan B.P.B.E Architecture Refactor

## 1. Objective / Context
As AutoLang expands its transpilation capabilities to support multiple language ecosystems (Rust via `a2rs`, C via `a2c`, Web via `a2vue`, Mobile via `a2jet`), the AutoMan build orchestrator needs a robust, scalable architectural model.
Historically, AutoMan supported building C projects via multiple backends (CMake, IAR, GHS, Ninja). Recently, it was established that `Ninja` should serve as the universal fast builder for C, and `a2rs` should be the default mixed project type.

This plan redesigns AutoMan around the **B.P.B.E** (Backend, Port, Builder, Export) architecture, providing a clean mental model and execution path for developers compiling Auto source code across distinct hardware and language ecosystems.

## 2. Current State & Problems
- **Ambiguous Terminology:** "Backend", "Env", "Port", and "Builder" are used interchangeably. "Ninja" and "CMake" are both considered backends, even though Ninja is a fast internal executor and CMake is an exportable project system.
- **Top-heavy CLI:** Commands needed to reflect every toolset variation, rather than relying on a smart configuration file.
- **Port Confusion:** The embedded `port` concept (Hardware/SDK/Toolset triple) overlaps confusingly with the concept of a transpilation target.

## 3. Proposed Design: The B.P.B.E. Architecture

AutoMan configuration (`pac.at`) and CLI will be strictly structured around this hierarchy:

### 3.1 `Backend` (Language / Ecosystem Destination)
The backend defines what language ecosystem AutoLang is transpiled into.
- **Values:** `c`, `rust`, `vue`, `jet`.
- **Scope:** Usually set once per project/application.
- **Example:** `backend: "c"` translates Auto to C logic.

### 3.2 `Port` (Target Environment & Hardware)
The port defines *for whom* the code is being compiled. A single `.at` codebase can support multiple ports.
- **Scope:** Defines the hardware architecture, OS, and requested Toolchain id.
- **C/Embedded (`a2c`):** E.g., `stm32` port combining `arm-cortex-m4`, and requesting the `arm-none-eabi-gcc` toolchain.
- **Rust (`a2rs`):** Maps to target triples like `x86_64-pc-windows-msvc`.
- **Vue (`a2vue`):** Maps to `web` or `desktop` (Tauri/Electron).

### 3.3 SDKs and Toolchains (Precision Semantics)
To keep projects portable across different developers and computers, we explicitly separate **SDKs** from **Toolchains**:

1. **SDKs (System Development Kits / HALs):**
   - An SDK (like STM32 HAL, or FreeRTOS ports) is inherently code that the project links against.
   - Therefore, an `sdk` is fundamentally a **Dependency** (formerly known as `device` in AutoMan).
   - Because an SDK is usually specific to a particular hardware target, **it is defined as a sub-node *inside* the `port` block**.
   - This ensures the SDK dependency is only downloaded and resolved when that specific `port` is active. It is NOT a global project dependency.

2. **Toolchains (Compilers & Linkers):**
   - A toolchain (like GCC, Clang, MSVC) has fixed binary paths (e.g., `CC`, `LINK`) specific to a developer's computer.
   - Therefore, physical toolchain paths and default flags MUST NOT be stored in `pac.at`.
   - `pac.at` only specifies the **Toolchain ID** that the port requires (e.g., `toolchain: "arm-none-eabi-gcc"`).
   - The actual implementation details (where `arm-none-eabi-gcc` exists on the disk) are defined in the developer's global environment settings (e.g., `~/.auto/am.at`).

### 3.4 `Builder` (Internal Orchestration Engine)
The builder is the tool AutoMan invokes internally to compile the transpiled code for a specific port. User rarely configurates this directly; AutoMan infers it from the Backend.
- **`c` Backend:** Defaults to **Ninja** (fast, universal).
- **`rust` Backend:** Defaults to **Cargo**.
- **`vue` Backend:** Defaults to **Vite** or **esbuild**.

### 3.5 `Export` (Foreign Project Generators)
Exports are third-party IDE and project files generated by AutoMan on demand, allowing developers to debug or compile outside the AutoMan ecosystem.
- **Values:** `cmake`, `iar`, `ghs`, `keil`.
- **Scope:** Configured per `port` (e.g., an IAR export only makes sense for an embedded STM32 port, not a Win32 port).

### 3.6 Example `pac.at` Syntax
```rust
name: "sensor_hub"
version: "1.0.0"
backend: "c" // The Backend

// The Win32 Port
port win32 {
    arch: "x86_64"
    os: "windows"
    toolchain: "msvc" // Uses global ~/.auto/am.at to find MSVC paths
    // Builder is Ninja by default. No exports needed.
}

// The Embedded Port
port stm32 {
    mcu: "stm32f407"
    
    // SDK Dependency is nested INSIDE the port
    sdk(stm32f4_hal, "1.0.5") {
        series: "f407"
    }
    
    toolchain: "arm-none-eabi-gcc" // Requests the cross-compiler from global settings
    
    // Feature: Export to IDEs
    exports: ["cmake", "iar"]
}
```

## 4. Execution Plan (Implementation Steps)

### Phase 1: Data Model & Schema Updates (`crates/auto-man`)
- [ ] Refactor `pac.at` parsing schema to enforce `backend` at the root project level.
- [ ] Refactor the `port` block schema to accept `arch`, `os`, `sdk`, `toolchain`, and an `exports` list.
- [ ] Default `a2c` builder to Ninja internally, removing CMake/IAR from the core "Builder" interfaces.

### Phase 2: Internal Builder Pipeline (`Builder` Trait)
- [ ] Define a clean internal `Builder` trait specifically for Ninja, Cargo, and Vite.
- [ ] Link `auto build` to strictly invoke the `Builder` determined by the `Backend`.

### Phase 3: Exporters Implementation (`Export` Trait)
- [ ] Move CMake, IAR, and GHS logic into an `Exporter` trait/module.
- [ ] Create CLI subcommands: `auto export --port <port_name> --format <export_name>` (e.g. `auto export --port stm32 --format iar`).

### Phase 4: CLI & Documentation Integration
- [ ] Update `auto.exe` CLI to route `--port` requests correctly (e.g. `auto build --port stm32`).
- [ ] Rewrite `auto-man` documentation to reflect the B.P.B.E architecture.
- [ ] Deprecate any leftover `am capp/clib` logic that violates the backend/port configuration model.

## 5. Testing Strategy
- **AST Tests:** Verify `pac.at` parses combinations of backend, ports, and exports correctly.
- **Integration Tests:** 
  - Run `auto build --port win32` and assert Ninja executes correctly for an `a2c` project.
  - Run `auto run` and assert Cargo executes correctly for an `a2rs` project.
  - Run `auto export --port stm32 --format cmake` and assert valid `CMakeLists.txt` generation.

## 6. Success Criteria
- [ ] `auto.exe` utilizes Ninja as the unquestioned internal builder for C.
- [ ] Projects can define multiple target environments using `port`.
- [ ] CMake, IAR, and GHS are structurally downgraded to "Exports" rather than primary Builders.
- [ ] `pac.at` natively separates Backend (rust/c), Port (stm32/win32), and Exports (cmake/iar).
